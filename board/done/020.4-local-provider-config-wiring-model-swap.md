# Local-provider config wiring + model swap

## Description
Wire local-provider selection and model/runtime settings into config and persisted settings so switching providers or swapping models is a Settings/config change, not a code change.

**Current state:** `.env` keys and settings persistence (016) already cover `AGENT_PROVIDER=llamacpp`, base URL, server path, model path, ctx size, GPU layers, and start mode. `resolveProviderRegistryConfig` hydrates agent calls at runtime. Catalog selection and download-applied paths are owned by **020.17–020.20**; this ticket covers the config contract those tickets write into.

## Acceptance Criteria
- [x] Provider selection supports choosing the local llama.cpp adapter from config/settings only, no code changes required
- [x] Model-specific settings (path, context, GPU layers) are read from config/settings and passed to the lifecycle manager
- [x] Invalid or missing required local-provider config returns typed validation errors at startup, distinguishing config problems from runtime/process problems
- [x] Unit tests verify provider and model swaps are honored through config-only changes (extend existing `settingsRuntime` / `selectProvider` coverage for path/ctx/gpu fields)
- [x] Persisted settings include fields (or stable keys) that 020.17–020.18 can set after catalog download (active model id, resolved `userData` model path, download state) without breaking existing manual-path saves
- [x] Documented config contract: happy-path users never need `.env`; `.env` remains override/dev fallback only

## Deferred (post-v1)
- Default model profile + override profile for per-role model selection (DM/NPC/lore) — one active model is enough until a second concurrent model is needed
