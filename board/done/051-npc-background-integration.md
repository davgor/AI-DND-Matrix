# EPIC: NPC background integration

Epic **050** gave player characters a preseeded background roster (Acolyte, Soldier, Noble, Isekai'd, etc. — `BACKGROUND_ROSTER`) plus an AI-written personal story, feeding the identity interview and opening scene as established fact. Epic **052** restructured NPC generation into a bundle-first (race, gender, alignment, class) pass followed by a context-grounded backstory pass. NPCs still have no equivalent of "what did this person do before" beyond whatever the backstory-writing step happens to invent — there is no structured background concept for them at all, and 050 explicitly scoped NPCs out ("player characters only in v1").

This epic closes that gap by **reusing 050's exact roster** — no new authoring, no duplicate descriptions — as a fifth silently-decided bundle field for NPCs, integrated directly into 052's two-phase pipeline: every speaking NPC also gets a `backgroundKey`, and that background's roster description becomes one more piece of established-fact grounding for the same backstory paragraph the NPC already receives.

**This is deliberately thinner than 050's player-facing feature.** NPCs get no onboarding page, no free-form story textarea, no Generate-with-guidance modal, and no separate stored "personal story" field — an NPC's entire personal narrative already lives in `npcs.backstory` (052's phase-2 output). Background here is just one more grounding input to that existing prose, exactly like gender/alignment/class blurbs already are.

**Land order matters more than file numbering here**: this epic depends on 050's roster (`BACKGROUND_ROSTER`) and on 052's phase-1/phase-2 pipeline and bulk-generation gender/class wiring. Build 050 and 052 first regardless of this file's number falling between them.

Broken down into sub-tickets **051.1–051.7**. This epic is done when all are complete and `npm test`, `npm run lint`, and `npm run build` pass.

## Target flow (extends 052's diagram)

```
Phase 1 — core bundle (052.5), now with one more field
  { canSpeak, temperament, race?, gender?, alignment?, class?, background? }
       │
       ▼
Phase 2 — final prompt (052.6), now with one more grounding input
  ... full race lore, gender blurb, alignment, class blurb,
  + the chosen background's roster description (BACKGROUND_ROSTER, static, no lock)
       │
       ▼
  { name, role, disposition, backstory }  → persisted NPC (backstory now reflects background)
```

Bulk campaign generation and additional-region generation (052.3's one-shot call) gain the same field, unchanged in shape — background is just a sixth thing requested per speaking NPC in that same JSON object.

## Core concepts

- **No new roster.** This epic reuses `BACKGROUND_ROSTER` (`src/engine/characterBackground/roster.ts`, 050.1) verbatim — same keys, same labels, same descriptions shown to players. An NPC background pick must be an exact key from that roster.
- **No lock, no per-campaign catalog, no personal story field.** Exactly like gender and class in 052, a background's description is static authored text used directly as prompt grounding — there is nothing to realize-once or reuse-after, and no `npcs.background_story` column. The NPC's one narrative field remains `npcs.backstory`.
- **`canSpeak`-gated, same as the other four bundle fields.** Non-speaking creatures never get a background, exactly like they never get race/gender/alignment/class.
- **NPC → party member promotion** (`confirmNpcPromotion`, 049.10's precedent) should carry `background_key` forward unchanged, the same way it already carries `race_key` forward — a promoted NPC becomes a player-kind character, and epic 050 already has a `background_key` column on `characters` waiting to receive it.

## Schema

One nullable, purely additive column (no CHECK constraint, no rebuild — same precedent as `race_key`/`gender_key`/`class_key`):

- `npcs.background_key TEXT` — set when `canSpeak` is true, `null` for non-speaking creatures and pre-epic NPCs.

## Definition of done

- Every speaking NPC generated by any path (bulk, additional-region, shortfall top-up, or the flagged single-NPC pipeline) has a `backgroundKey` set from `BACKGROUND_ROSTER`; non-speaking NPCs have `null`
- The flagged-NPC pipeline's phase-1 core bundle includes background selection; phase-2's final backstory prompt is grounded in the chosen background's roster description as established fact
- NPC → party member promotion carries `background_key` forward unchanged onto the new character row (no regeneration, no lore step — there is none to run)
- NPC traits UI shows the NPC's background label alongside gender/class/alignment
- Smoke runbook covers a flagged NPC whose backstory visibly reflects its chosen background (e.g. a "Soldier" NPC's generated backstory references military service)

## Relationship to other epics

- **050** (character backgrounds): hard dependency — this epic reuses `BACKGROUND_ROSTER` (050.1) unchanged. No new authoring here.
- **052** (NPC core identity bundle): hard dependency — this epic extends `NpcCoreBundle`, `buildNpcCoreBundlePrompt`/`generateNpcCoreBundle` (052.5), `buildFlaggedNpcFinalPrompt`/`generateFlaggedNpcDetails` (052.6), `generateFlaggedNpc` (052.7), the bulk/additional-region prompt+normalize changes (052.3), and the NPC traits UI (052.8) — all of it must exist first.
- **049** (race selection / NPC promotion): `confirmNpcPromotion`'s race-forwarding behavior (049.10) is the direct precedent for this epic's background-forwarding ticket (051.6).

## Out of scope

- Any change to `BACKGROUND_ROSTER` itself, or to the player-facing background selection page — both are owned entirely by epic 050
- A separate "personal story" field for NPCs, a Generate-with-guidance-prompt flow for NPCs, or any NPC-facing UI step — NPC background is silent grounding only, folded into the existing backstory prose
- Mechanical effects of background on any character
- Backfilling `background_key` onto NPCs that existed before this epic
- Custom/player-minted backgrounds for NPCs (the player-facing screen has none either; out of scope there too)
