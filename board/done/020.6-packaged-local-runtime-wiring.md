# Packaged local runtime + userData asset layout

## Description
Make packaged builds run the local-provider path reliably for end users: resolve runtime binary and model paths from **persisted Settings** and a stable **`userData` asset layout** (downloaded models / acquired runtime), validate at startup/Apply, and surface clear recovery hints when assets are missing.

**Product constraint (rescoped):** the installer / portable `.exe` stays lean — it does **not** embed multi-GB `.gguf` weights. Models (and optionally the runtime binary) are fetched on demand into Electron `userData` via 020.18–020.19. Packaged path resolution must work for those downloaded assets and for advanced manual absolute paths.

## Acceptance Criteria
- [x] Packaged app can launch or attach to the configured local llama.cpp runtime in a production install, not only in dev mode
- [x] Runtime/model path resolution works in packaged environments using persisted settings, preferring the documented `userData` layout for catalog downloads (absolute paths for advanced BYO still allowed)
- [x] Missing runtime, incomplete download, or invalid runtime/model path surfaces a clear typed error with a recovery hint (open Settings → retry download / acquire runtime)
- [x] Packaging notes document: lean installer, `userData` asset directories, reference catalog model, VRAM/RAM assumptions, and advanced BYO / attach fallback
- [x] Unit or integration tests cover path resolution for `userData`-relative downloaded assets vs absolute manual paths

## Notes
- Download implementation belongs in 020.18–020.19; this ticket owns layout + packaged resolution + docs.
- Manual generation proof belongs in 020.7.
- If SQLite/native-module issues appear only in packaged Electron, follow `npm run rebuild:electron` guidance from `docs/runbooks/startup-smoke-test.md`.
