# 112.3 Persist usage events

## Description
Store usage events in SQLite (app-level and/or per-campaign) so costs can be aggregated across sessions without relying on memory or provider dashboards.

## Acceptance Criteria
- [ ] Schema/migration adds an `llm_usage_events` (name flexible) table with indexes on purpose, bucket, campaign_id, and created_at
- [ ] Repository API can insert events and query by time range / campaign / purpose / bucket
- [ ] Delete-campaign path cascades or orphans are defined (prefer cascade with campaign when campaign-scoped)
- [ ] Recording is best-effort: a metering write failure must not fail the player-facing generate result (log + continue)
- [ ] Tests cover insert, query aggregation seeds, and campaign delete behavior
