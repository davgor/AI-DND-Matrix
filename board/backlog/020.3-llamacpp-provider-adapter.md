# llama.cpp provider adapter implementation

## Description
Implement the llama.cpp provider adapter against `POST /v1/chat/completions`, satisfying the same shared provider interface the Claude adapter implements. Process-readiness is handled at boot by the lifecycle manager (020.2) and epic 015 — not duplicated inside every `generate()` call for v1.

**Current state:** `createProviderRegistry` routes `llamacpp` to `createPlayer2Provider({ baseUrl })` because both speak the same OpenAI-compatible chat-completions API. This scaffold is sufficient for early wiring but is not the final adapter.

**Recommended implementation:** extract a shared `createOpenAiChatCompletionsProvider({ baseUrl, unreachableErrorClass, requestErrorClass })` used by Player2 and llama, or add a thin `createLlamaCppProvider` wrapper with llama-specific error types and log redaction.

Like every provider adapter, it returns generated text only — it never mutates game state itself.

## Acceptance Criteria
- [ ] `llamacpp` mode uses a dedicated adapter (or shared OpenAI-chat adapter), not a silent alias to the Player2 module
- [ ] The adapter maps provider call input to llama.cpp's chat-completions request shape and maps the response back to the shared provider output shape (`system` + `user` messages, `max_tokens`)
- [ ] Malformed/missing response fields return a typed adapter parse error, never an uncaught exception
- [ ] No secrets or sensitive local paths/prompt content are logged in plain text from adapter error paths
- [ ] Unit tests with a mocked local HTTP layer cover a successful generation path and a malformed-output path

## Deferred (not required for epic close)
- Per-request lifecycle gate when manager state is `degraded` — only add if smoke shows calls after unexpected process exit
- llama.cpp `response_format` / JSON-schema guided output — agents already use prompt + `tryParseJson` + retry loops (same as Player2/Claude today)
