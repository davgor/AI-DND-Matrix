# m005.1 Remote-play architecture and authority model

## Description
Define the PC-host-authoritative remote-play architecture: what runs on the PC, what the thin remote client may do, and how this differs from cloud-hosted MCP Apps or multiplayer guests.

## Acceptance Criteria
- [ ] Authority model is documented: PC owns engine, SQLite, agents, and provider calls; remote client is non-authoritative UI
- [ ] Explicit list of allowed remote→host commands (e.g. player action, sheet queries) vs forbidden operations (direct provider calls, raw SQL, arbitrary IPC)
- [ ] Failure domains are identified (PC sleep, app quit, network partition, provider down)
- [ ] Typed protocol contract outline exists for session, action, stream, and state-sync message categories
- [ ] Non-goals are explicit: no developer-hosted LLM; no public unauthenticated Player2 proxy
