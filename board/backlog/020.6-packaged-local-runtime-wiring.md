# Packaged local runtime wiring

## Description
Make packaged builds able to run the local-provider path reliably for downloaded users: resolve runtime binary/model paths from persisted settings (not dev-directory assumptions), validate them at startup, and surface clear, actionable errors when they're missing.

**Feasibility:** the app does **not** ship `llama-server` or a `.gguf` inside the portable `.exe`. Users install the runtime separately and configure paths in Settings — same as dev. "Packaged" here means the production Electron binary + user-data settings store, not an embedded inference bundle.

## Acceptance Criteria
- [ ] Packaged app can launch or attach to the configured local llama.cpp runtime in a production install, not only in dev mode
- [ ] Runtime/model path resolution works in packaged environments using persisted settings and absolute paths chosen by the user
- [ ] Missing runtime or invalid runtime/model path surfaces a clear typed error with a recovery hint (link to install instructions / Settings)
- [ ] Packaging notes document expected user prerequisites: runtime binary install (`winget install llama.cpp` or GitHub release), model placement, reference model suggestion, minimum GPU/RAM assumptions

## Notes
- Manual generation proof belongs in 020.7, not duplicated here.
- If SQLite/native-module issues appear only in packaged Electron, follow `npm run rebuild:electron` guidance from `docs/runbooks/startup-smoke-test.md`.
